Módulos del sistema

8 módulos que trabajan juntos para cubrir el proceso completo de inspección

🔐 Autenticación
Login con usuario + PIN hasheado. Rate limiting de 5 intentos, sesión con token de 24h, doble canal cookie + localStorage.

📋 Inspección preoperacional
Formulario con todos los ítems del vehículo (llantas, frenos, luces, documentos). El conductor firma digitalmente en el campo canvas.

📄 Generación de PDF
Al enviar el formulario, WeasyPrint toma la plantilla HTML, inyecta los datos y genera el PDF con logo, fuentes Futura y firma del conductor.

📊 Historial del conductor
Cada conductor ve únicamente sus propias inspecciones. Puede descargar PDFs individuales o un consolidado de sus últimas 15 inspecciones.
routes_inspeccion.py
👑 Panel administrador
El admin ve todas las inspecciones de todos los conductores, gestiona usuarios, cambia PINs y descarga cualquier reporte.
routes_admin.pyadmin/
📑 Reportes consolidados
Al acumular 15 inspecciones, se genera un PDF consolidado, las filas se archivan en reportes_inspeccion y se liberan de la tabla activa.

El admin crea, edita y desactiva usuarios. Puede asignar rol conductor o admin. El campo activo=0 bloquea el acceso aunque haya token válido.
routes_admin.py
📝 Auditoría
Cada acción del admin (crear/editar/eliminar) queda registrada automáticamente en logs_auditoria. No se puede borrar desde la interfaz.

Capa 1 — Frontend (Navegador)
Plantillas HTML renderizadas por Jinja2 en el servidor. El CSS usa el tema incubant-theme.css. El JS maneja la firma canvas, el envío del formulario vía fetch() y la gestión del token en localStorage. No hay framework SPA — Jinja2 es suficiente para un sistema interno.
# Archivos principales del frontend
templates/login.html
templates/form.html # Formulario de inspección + canvas firma
templates/lista_inspecciones.html
templates/admin/dashboard.html
static/incubant-theme.css
Capa 2 — Backend (FastAPI + Uvicorn)
FastAPI recibe cada petición HTTP, valida autenticación, ejecuta la lógica de negocio y devuelve la respuesta. Uvicorn es el servidor ASGI que lo corre. Los routers dividen la lógica en módulos: auth, inspecciones, admin.
app/main.py # Punto de entrada, registra routers
app/security.py # Token, login, rate limiting
app/routes_inspeccion.py # CRUD inspecciones + PDF
app/routes_admin.py # Gestión usuarios + auditoría
app/utils_pdf.py # WeasyPrint → PDF
Capa 3 — Datos (SQLAlchemy + MySQL)
SQLAlchemy ORM traduce Python a SQL. Los modelos en models.py definen las 4 tablas. database.py gestiona la conexión y la inyección de sesiones con get_db().
app/database.py # engine, SessionLocal, get_db()
app/models.py # Usuario, Inspeccion, Reporte, Log
Capa 4 — Disco (archivos generados)
Los PDFs y firmas se guardan en disco, no en la BD. Los binarios grandes en base de datos la hacen lenta. La BD solo guarda el nombre del archivo — la ruta real es predecible.
app/data/generated_pdfs/ # PDFs de inspecciones individuales
app/data/firmas/ # PNGs de firmas digitales de conductores
app/static/img/logo.png # Logo para el PDF (nombre en minúscula)

Herramientas y stack

Cada tecnología elegida por una razón específica

⚡ FastAPI Backend
El cerebro del sistema. Recibe cada petición HTTP, valida los datos automáticamente con Pydantic, ejecuta la lógica y devuelve la respuesta. Tiene documentación automática en /docs.

¿Por qué no Django? — FastAPI es más rápido para APIs modernas, más ligero, y la validación automática de datos evita errores de tipo.
🖨️ WeasyPrint PDF
Convierte HTML+CSS a PDF en el servidor. Le das una plantilla Jinja2 con los datos del conductor y devuelve un PDF con logo, fuentes Futura y firma. La analogía exacta: "Ctrl+P en Chrome, pero automático en el servidor."

¿Por qué no ReportLab? — WeasyPrint usa CSS normal. El diseñador puede editar la plantilla sin saber Python.
🗂️ SQLAlchemy ORM
Permite hablar con MySQL escribiendo Python en vez de SQL. Los modelos en models.py definen las tablas y las relaciones. Las tablas se crean automáticamente al iniciar.
🎨 Jinja2 Templates
Motor de plantillas HTML. Toma archivos con variables como {{ nombre_conductor }} y los reemplaza con datos reales antes de enviarlos al navegador.
🔒 Passlib + bcrypt Auth
Hashea los PINs de forma irreversible. "1234" se convierte en $2b$12$eIW.... Ni el desarrollador puede recuperar el PIN original.
⚙️ python-dotenv Config
Lee el archivo .env para las variables de configuración (contraseñas, host de BD). Las credenciales nunca están en el código fuente.
🚀 Uvicorn + Gunicorn Servidor
Uvicorn es el servidor ASGI que corre FastAPI. En producción, Gunicorn gestiona múltiples workers de Uvicorn para manejar más tráfico.
🌐 Nginx Proxy
Proxy reverso que recibe el tráfico en el puerto 80/443 y lo pasa al puerto 8000 de Uvicorn. También sirve los archivos estáticos eficientemente.

⚠️ Reglas críticas de archivos
Logo en minúscula: el archivo debe llamarse exactamente logo.png — WeasyPrint en Linux es case-sensitive. Si se llama Logo.png, el PDF no incluirá el logo.

Nunca subir .env: contiene la contraseña de la base de datos. Usa .env.example como referencia en cada nuevo servidor.

Nunca subir venv/: el entorno virtual tiene binarios específicos del OS. Siempre recrearlo con pip install -r requirements.txt.

Seguridad — 7 capas

Cada capa protege contra un vector de ataque diferente

🔐
bcrypt — PINs irreversibles
El PIN nunca se guarda en texto plano. bcrypt genera un hash de 60 caracteres con sal aleatoria. Ni el admin ni el desarrollador pueden recuperar el PIN original. Si el sistema es comprometido, los PINs siguen seguros.
⏱️
Rate limiting — 5 intentos / 5 minutos por IP
Si una IP falla el login 5 veces en 5 minutos, queda bloqueada. Previene ataques de fuerza bruta donde alguien intenta adivinar el PIN probando combinaciones automáticamente.
🎫
Tokens con expiración de 24 horas
El token se genera en el login y expira automáticamente. El logout no solo cierra la sesión en el navegador — invalida el token en la BD de verdad, por lo que no puede ser reutilizado.
🍪
Doble canal de autenticación
El token se entrega por dos canales: (1) cookie httpOnly — el navegador la envía automáticamente, el JS no puede leerla (anti-XSS); (2) JSON → JS en localStorage para peticiones fetch(). La cookie sirve para acceder por URL directa, el localStorage para las llamadas AJAX.
👤
Separación estricta de roles
Un conductor solo puede ver sus propias inspecciones — la consulta SQL filtra por usuario_id. El rol admin ve todo. No hay forma de escalar privilegios desde la interfaz.
🚫
Handler global 401/403
Si alguien intenta acceder a una URL protegida sin token o con token inválido, el sistema redirecciona automáticamente al login. No hay páginas expuestas accidentalmente.
🔕
Registro deshabilitado
No existe un formulario público de registro. Los usuarios solo pueden ser creados por el administrador. Esto impide que personas no autorizadas creen cuentas en el sistema.
# Flujo de autenticación completo
1. POST /auth/login → recibe usuario + PIN
2. Rate limiting check → ¿5 fallos de esta IP en 5 min? → bloqueo
3. bcrypt.verify(PIN, pin_hash) → ¿coincide?
4. Genera token aleatorio → se guarda en BD con expiración
5. Respuesta: cookie httpOnly + JSON token
6. Cada petición: get_current_user() → lee token → BD → ¿expiró? ¿activo=1?

Guía de despliegue — Hostinger VPS

12 fases desde cero hasta el sistema en producción

FASE 1 Conectarse al VPS por SSH
ssh root@IP_DE_TU_VPS
La IP y contraseña están en hPanel → VPS → Administrar
FASE 2 Actualizar servidor e instalar dependencias
apt update && apt upgrade -y

apt install -y python3.11 python3.11-venv python3-pip \
  mysql-server nginx git curl \
  libcairo2 libpango-1.0-0 libpangocairo-1.0-0 \
  libgdk-pixbuf2.0-0 libffi-dev shared-mime-info libharfbuzz0b
⚠️ Las librerías libcairo2 y libpango son obligatorias para WeasyPrint. Sin ellas los PDFs no se generan.
FASE 3 Configurar MySQL
mysql_secure_installation
# Responde: Y Y Y Y Y — ponle contraseña al root

mysql -u root -p

CREATE DATABASE misionales CHARACTER SET utf8mb4;
CREATE USER 'misionales_user'@'localhost' IDENTIFIED BY 'Password123!';
GRANT ALL PRIVILEGES ON misionales.* TO 'misionales_user'@'localhost';
FLUSH PRIVILEGES; EXIT;
FASE 4 Clonar el repositorio
cd /home
git clone https://github.com/Sant-Beco/Misionales.git misionales
cd misionales
FASE 5 Entorno virtual + dependencias
python3.11 -m venv venv
source venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
pip install gunicorn
FASE 6 Crear el archivo .env
nano .env

# Contenido del archivo:
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=misionales
DB_USER=misionales_user
DB_PASSWORD=Password123!
SECRET_KEY=clave_larga_y_aleatoria_aqui
ENVIRONMENT=production

# Guardar: Ctrl+O → Enter → Ctrl+X
FASE 7 Migración de base de datos
mysql -u misionales_user -p misionales < migration.sql
FASE 8 Prueba manual — verificar que funciona
source venv/bin/activate
uvicorn app.main:app --host 0.0.0.0 --port 8000
Abre en el navegador: http://IP_DE_TU_VPS:8000
Si ves el login ✅ — detén con Ctrl+C y continúa.
FASE 9 Configurar systemd (servicio permanente)
nano /etc/systemd/system/misionales.service

[Unit]
Description=Misionales FastAPI
After=network.target mysql.service

[Service]
User=root
WorkingDirectory=/home/misionales
ExecStart=/home/misionales/venv/bin/gunicorn app.main:app \
  -w 2 -k uvicorn.workers.UvicornWorker \
  --bind 127.0.0.1:8000 --timeout 120
Restart=always
EnvironmentFile=/home/misionales/.env

[Install]
WantedBy=multi-user.target
systemctl daemon-reload
systemctl enable misionales
systemctl start misionales
systemctl status misionales # debe mostrar "active (running)"
FASE 10 Configurar Nginx (proxy reverso)
nano /etc/nginx/sites-available/misionales

server {
  listen 80;
  server_name IP_DE_TU_VPS;
  client_max_body_size 20M;

  location / {
    proxy_pass http://127.0.0.1:8000;
    proxy_set_header Host $host;
    proxy_read_timeout 120s;
  }
  location /static {
    alias /home/misionales/app/static;
  }
}
ln -s /etc/nginx/sites-available/misionales /etc/nginx/sites-enabled/
rm /etc/nginx/sites-enabled/default
nginx -t # debe decir "syntax is ok"
systemctl restart nginx
FASE 11 Crear primer administrador
cd /home/misionales
source venv/bin/activate
python admin_cli.py
# Sigue las instrucciones para crear el usuario admin
FASE 12 Verificación final y comandos del día a día
# Verificar que todo funciona
systemctl status misionales # debe decir active (running)
systemctl status nginx
journalctl -u misionales -f # logs en tiempo real

# Actualizar código desde GitHub
